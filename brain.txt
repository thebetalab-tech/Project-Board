## 📖 ABSTRACT (why this exists)

Students currently have to visit their mentor multiple times in person just to explain/pitch project ideas. Groups are formed over WhatsApp. Technology selection happens via Google Forms. Faculty struggle to manage multiple Excel sheets just to track who is doing what and to prevent duplicate projects across the same batch.

**Project Board fixes all of this in one platform:**
- Students form groups, pick tech, choose a mentor — all online
- Faculty get a dashboard with pending requests, proposals, and a built-in duplicate checker
- Admin manages the whole system: creates faculty accounts, adds technologies, assigns faculty

---

## 👥 USER ROLES (3 roles, 1 Users table)

### Student / Group Leader
- Self-registers (email + password + enrollment number)
- Creates a group → becomes Leader automatically
- Invites up to 4 other students (max 5 per group including leader)
- Selects technology stack
- Selects a faculty mentor (filtered by their chosen tech)
- Submits project idea: Title, Type (IDP/UDP), Functionality
- Sees duplicate warning before submitting
- Can resubmit if rejected

### Faculty / Mentor
- Account created by Admin only — NO self-registration
- Assigned to one or more technologies by Admin
- Receives group requests → Approve or Reject
- Receives project proposals → Reviews duplicate panel → Approve or Reject
- After approving: adds domain keyword tags to the project
- Can Drop active groups
- Can Generate PDF report of their technology's groups and projects

### Admin
- Pre-seeded in DB. Single account. Full control.
- Creates faculty accounts
- Adds / edits / removes technologies
- Assigns technologies to faculty
- Views all students, groups, projects system-wide
- Can force-approve or force-reject anything
- Views audit log

## 🔄 COMPLETE SYSTEM FLOW (step by step)

```
Student Registers
    ↓
Creates Group → becomes Leader (Status: Forming)
    ↓
Invites up to 4 members (they accept/reject)
    ↓
Selects Technology (from admin-managed list)
    ↓
Selects Mentor (faculty filtered by chosen tech)
    → Sends Group Request to Faculty (Status: Pending)
    ↓
Faculty sees request on dashboard
    → Approves Group (Status: Active)  OR  Rejects (back to Forming)
    ↓
[Only after group is Active]
Leader submits Project Idea
    → System runs duplicate check (Jaccard similarity + keyword tags)
    → If similar found: WARNING shown — leader must tick "I understand" to proceed
    → Proposal sent to faculty (Status: Pending)
    ↓
Faculty opens proposal
    → Sees duplicate similarity panel AGAIN (independent server-side check)
    → Approves with remark  OR  Rejects with reason
    ↓
[If Approved]
Faculty adds domain keyword tags (e.g. hospital, patient, clinic)
These tags protect against future renamed duplicates
    ↓
Project locked. Group's idea officially registered.
    ↓
[If Rejected]
Leader can submit a new idea — cycle repeats
```

## 🔑 DUPLICATE PROJECT DETECTION (the core feature)

> **Problem:** Two groups submit "Hospital Management System" and "HMS – Hospital Software" for PHP. Simple equality check misses this.

### 4-Layer Defense System

|
 Layer 
|
 Who Sees It 
|
 Can Student Bypass? 
|
|
---
|
---
|
---
|
|
 Normalize + Jaccard on student submit 
|
 Student gets warning modal 
|
 YES — advisory only 
|
|
 Same check shown to faculty independently 
|
 Faculty sees it when reviewing 
|
 HARD — faculty knows domain 
|
|
 Keyword tag match (semantic check) 
|
 Both student and faculty 
|
 NO — tags set by faculty, invisible to students 
|
|
 Admin audit log 
|
 Admin reviews approvals 
|
 NO — final safety net 
|

### How Normalization Works
```
"Hospital Management System" → strip stopwords → "hospital"
"Hospital Software Portal"   → strip stopwords → "hospital"
Both match → 100% Jaccard similarity
```

Stopwords stripped: system, management, app, application, project, software, platform, portal, web, the, a, of, for, and

### Keyword Tag Bypass Prevention
When Group A gets "Hospital Management System" approved in PHP, faculty tags:
`hospital, patient, ward, clinic, doctor`

When Group B submits "Clinic Patient Record System" (PHP):
- "clinic" and "patient" are in the existing tags
- System fires warning **automatically** even though "Hospital" is not in the new name

## 🗄️ DATABASE SCHEMA (SQL Server — 7 Tables)

### Users
```sql
UserId       INT IDENTITY PK
FullName     NVARCHAR(100)
Email        NVARCHAR(100) UNIQUE
Password     NVARCHAR(256)        -- SHA-256 + salt, never plain text
EnrollmentNo NVARCHAR(20) NULL    -- students only
Role         NVARCHAR(10)         -- 'Student', 'Faculty', 'Admin'
CreatedAt    DATETIME DEFAULT GETDATE()
```

### Technologies
```sql
TechId    INT IDENTITY PK
TechName  NVARCHAR(50) UNIQUE     -- 'PHP', '.NET', 'Python', 'Java', 'React', 'Android'
```

### Faculty *(junction: which faculty handles which tech)*
```sql
FacultyId  INT PK, FK → Users(UserId)
TechId     INT PK, FK → Technologies(TechId)
```

### Groups
```sql
GroupId    INT IDENTITY PK
GroupName  NVARCHAR(100)
LeaderId   INT FK → Users(UserId)
TechId     INT FK → Technologies(TechId)
MentorId   INT NULL, FK → Users(UserId)
Status     NVARCHAR(30) DEFAULT 'Forming'
           -- Options: 'Forming', 'Pending', 'Active', 'Dropped'
```

### GroupMembers *(junction: who is in which group)*
```sql
GroupId    INT PK, FK → Groups(GroupId)
UserId     INT PK, FK → Users(UserId)
JoinStatus NVARCHAR(15) DEFAULT 'Pending'
           -- Options: 'Pending', 'Accepted', 'Rejected'
```

### Projects
```sql
ProjectId       INT IDENTITY PK
GroupId         INT FK → Groups(GroupId)
ProjectType     CHAR(3)              -- 'IDP' or 'UDP'
ProjectTitle    NVARCHAR(150)
NormalizedTitle NVARCHAR(150)        -- auto-computed, stopwords stripped
Functionality   NVARCHAR(1000)
Status          NVARCHAR(15) DEFAULT 'Pending'
                -- Options: 'Pending', 'Approved', 'Rejected'
SubmittedAt     DATETIME DEFAULT GETDATE()
```

### ProjectKeywords *(domain tags for duplicate detection)*
```sql
TagId      INT IDENTITY PK
ProjectId  INT FK → Projects(ProjectId)
Keyword    NVARCHAR(30)
```

- `--orange` = primary action buttons only (CTAs). Rare.
- `--clay` = rejection/error states. Rare.
- `--aqua` = approval/success/available. Positive feedback only.

## 📌 IMPORTANT DECISIONS (don't change these)

1. **Faculty cannot self-register.** Admin creates all faculty accounts. This is intentional — prevents fake mentor accounts.
2. **5-member group cap is hard.** Enforced in both UI and stored procedure. No exceptions.
3. **Duplicate check is two-step.** Check button and Submit button are separate — leader must explicitly see results first.
4. **Faculty is the real gatekeeper.** Student-side duplicate warning is advisory. Faculty-side check is independent and unavoidable.
5. **Keyword tags are faculty-only.** Students cannot see or manipulate keyword tags. Only faculty and admin can add/edit them.
6. **Technology lock.** Cannot change technology after selecting a mentor (TechId in Groups table updates once, no update allowed after MentorId is set).
7. **Cascade deletes.** GroupMembers cascade on GroupId delete. ProjectKeywords cascade on ProjectId delete. Historical records (AuditLog) do NOT cascade.
